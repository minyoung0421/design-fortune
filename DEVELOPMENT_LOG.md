### Commit 1–3: 프로젝트 부트스트랩 (2026-03-10)

**커밋:** `8ce0863` → `5e2c004` → `6e781b6`

#### 문제 정의

프로젝트를 시작할 때 가장 먼저 맞닥뜨린 문제는 "무엇을 만들 것인가"보다 "어떤 기반 위에 만들 것인가"였다. 기존의 Figma-to-Code 워크플로우는 디자이너가 Figma에서 값을 정의하면, 개발자가 그 값을 직접 보고 코드에 하드코딩하는 수동 방식이었다. 이 방식은 다음 문제를 반복적으로 발생시킨다:

- Figma에서 색상이 바뀌면 코드베이스 전체를 검색해서 수동 수정
- 팀원마다 다른 방식으로 스타일을 기술하여 일관성 붕괴
- AI 에이전트가 디자인 원칙 없이 임의 값으로 코드를 생성

이 세 가지 문제를 동시에 해결할 아키텍처가 필요했다.

#### 기술적 결정

**Bun 채택:** Node.js 대신 Bun을 런타임으로 선택한 이유는 단순한 속도 문제가 아니다. Bun은 패키지 매니저(`bun install`), 번들러(`bun build`), 테스트 러너를 단일 바이너리로 통합한다. MCP 서버처럼 실시간 WebSocket 통신이 필요한 환경에서 Cold-start 지연을 최소화하는 것이 핵심이었고, Bun의 네이티브 TypeScript 지원은 별도 `ts-node` 설정 없이 개발 속도를 높였다.

**TypeScript 채택:** 디자인 토큰은 전체 코드베이스에서 참조되는 중앙 데이터다. `tokens.ts`의 타입이 보장되지 않으면, 존재하지 않는 토큰 키를 참조하는 버그가 런타임에만 드러난다. TypeScript의 `as const` 타입 추론으로 `tokens.colors.accent`처럼 잘못된 키 접근을 컴파일 타임에 차단한다.

#### 결과

기본 프로젝트 구조와 저장소가 생성되었다. 이 단계에서 중요한 것은 코드보다 "방향성 합의"였다 — 모든 팀원과 AI 에이전트가 동일한 원칙 위에서 작업하도록 토대를 마련하는 것.

### Commit 4: 핵심 결과물 통합 (2026-03-13)

**커밋:** `0f76ad0`

#### 문제 정의

초기 프로토타입과 실제 제출용 결과물 사이의 격차를 좁혀야 했다. 개념 증명(PoC) 수준의 코드가 실제 심사 기준을 충족하려면 구조적인 완성도가 필요했다.

#### 기술적 결정

**MCP(Model Context Protocol) 채택:** 이 프로젝트에서 MCP는 단순한 API 연결 레이어가 아니다. MCP를 선택한 핵심 논리는 다음과 같다:

1. **표준화된 컨텍스트 전달:** AI 에이전트(Cursor)가 Figma의 디자인 데이터를 읽을 때, REST API는 요청-응답 구조로 매번 컨텍스트를 재구성해야 한다. MCP는 지속적인 컨텍스트 세션을 유지하므로, 에이전트가 Figma 레이어 구조와 디자인 토큰을 대화 흐름 안에서 지속적으로 참조할 수 있다.

2. **도구 인터페이스 표준화:** `get_document_info`, `get_selection` 같은 MCP 도구를 통해 AI 에이전트는 Figma를 마치 코드 파일처럼 "읽는다". 이 추상화 덕분에 에이전트는 Figma API의 복잡한 데이터 구조를 직접 파싱하지 않아도 된다.

3. **WebSocket 실시간 통신:** `socket.ts`가 WebSocket 서버로 동작하여 Figma 플러그인과 Cursor 에이전트 사이에 실시간 채널을 형성한다. 디자이너가 Figma에서 컴포넌트를 선택하는 순간, 에이전트는 즉시 그 컨텍스트를 받아 코드 생성에 활용할 수 있다.

#### 결과

MCP 서버, WebSocket 소켓, Figma 플러그인 연동이 통합된 핵심 결과물이 완성되었다.

### Commit 5: 디자인 시스템 토큰 체계 구축 (2026-03-14)

**커밋:** `a90c555`

#### 문제 정의

코드가 완성된 시점에도 "디자인 원칙이 코드에 어떻게 반영되었는가"를 설명하는 문서가 없었다. AI 에이전트에게 디자인 원칙을 주입할 메커니즘도 부재했다. 컴포넌트를 만들 때마다 색상값을 직접 입력하는 패턴이 반복될 위험이 있었다.

#### 기술적 결정

**`src/styles/tokens.ts` 분리 이유:**

디자인 토큰을 별도 파일로 분리한 것은 단순한 정리 차원이 아니다. 이 결정에는 세 가지 아키텍처적 논리가 있다.

첫째, **변경 전파의 단일화.** 색상 하나가 바뀔 때 영향 범위를 `tokens.ts` 파일 한 줄로 제한한다. 컴포넌트 파일에 색상이 흩어져 있으면 하나의 변경이 수십 개의 파일 수정을 요구한다.

```typescript
// 이 한 줄을 바꾸면 accent 색상을 사용하는 모든 컴포넌트에 즉시 반영된다
accent: '#3182CE',  →  accent: '#2B6CB0',
```

둘째, **AI 에이전트의 참조 범위 제한.** `DRAGME.md`에서 AI 에이전트에게 "모든 수치는 `tokens.ts`에서만 참조하라"고 명시함으로써, 에이전트가 생성하는 코드의 하드코딩 발생률을 구조적으로 0에 가깝게 수렴시킨다.

셋째, **TypeScript `as const`를 통한 타입 안전성.** 토큰 객체가 타입으로 추론되므로 `tokens.colors.nonExistent`처럼 잘못된 참조는 컴파일 단계에서 오류로 잡힌다.

**`DRAGME.md` 구성 이유:**

AI 에이전트는 매 대화마다 새로운 컨텍스트에서 시작한다. `DRAGME.md`는 에이전트가 프로젝트를 열 때 최우선으로 읽어야 할 "개발 원칙 주입 파일"로 설계되었다. 규칙을 매번 채팅으로 전달하는 대신, 파일 하나로 에이전트의 행동 원칙을 고정한다.

#### 결과

- `src/styles/tokens.ts`: 색상 4종, 간격 4단계, 타이포그래피 2계층, 보더 라디우스 3단계 정의
- `DRAGME.md`: AI 에이전트 개발 원칙 문서화
- `DESIGN_SYSTEM.md`: 시스템 전체 스펙 문서화
- `readme.md`: 프로젝트 전체 개요 재작성

### Commit 7: Atomic 컴포넌트 확장 (2026-03-14)

**커밋:** `7732ba9`

#### 문제 정의

심사에서 "컴포넌트 수가 적다"는 지적을 받았다. Button 하나만으로는 디자인 시스템의 실효성을 증명하기 어렵다. 서로 다른 역할의 컴포넌트가 동일한 토큰 시스템 위에서 일관되게 작동한다는 것을 보여야 했다.

#### 기술적 결정

`Input`과 `Badge`를 추가 컴포넌트로 선택한 이유:

- **Input**: 에러 상태(`tokens.colors.error`), 비활성 상태, `aria-invalid` 등 토큰이 단순 색상 외 상태(state) 표현에도 활용됨을 증명
- **Badge**: `variant` prop으로 4가지 시멘틱 컬러(`primary/accent/success/error`)를 모두 사용 — 토큰 시스템의 시멘틱 일관성을 단 하나의 컴포넌트로 시각화

두 컴포넌트 모두 `tokens.ts`의 모든 카테고리(colors, spacing, typography, borderRadius)를 빠짐없이 참조하도록 설계했다.

#### 결과

- 3개 컴포넌트(Button, Input, Badge)가 동일한 토큰 기반 위에서 일관되게 작동
- 하드코딩 수치 0건 달성
- 모든 인터랙티브 요소에 ARIA 속성 포함

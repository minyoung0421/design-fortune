# Testing & Quality Assurance

Design Fortune의 품질 기준은 **디자인-코드 일치도**와 **사용자 경험의 일관성**입니다.
이 문서는 사용성 테스트 계획과 디자인 검토 프로세스를 정의합니다.

---

## 사용성 테스트 계획

### 목표

실제 사용자(디자이너 및 프론트엔드 개발자)가 토큰 기반 컴포넌트를 적용할 때의 경험을 검증하고, 디자인 시스템의 실효성을 수치로 확인합니다.

### 핵심 성공 지표 (KPI)

| 지표 | 목표 기준 |
|------|-----------|
| Figma-코드 시각적 일치도 | 95% 이상 |
| 토큰 미적용 컴포넌트 비율 | 0% (하드코딩 0건) |
| 신규 컴포넌트 작성 시간 | 기존 대비 40% 단축 |
| 접근성 위반 항목 | 0건 |

---

### 테스트 라운드 1: 컴포넌트 적용성 검증

**대상:** 프론트엔드 개발자 3~5인
**방법:** Think-aloud 프로토콜 (작업하면서 생각을 소리내어 말하기)

**시나리오:**
1. `tokens.ts`만 보고 Button 컴포넌트를 직접 작성하기
2. 기존 하드코딩된 컴포넌트를 토큰 기반으로 리팩토링하기
3. AI 에이전트(`DRAGME.md` 주입)에게 새 컴포넌트 생성 요청하기

**피드백 수집 항목:**
- 토큰 네이밍이 직관적인가?
- `tokens.ts` 구조에서 원하는 값을 빠르게 찾을 수 있는가?
- AI가 생성한 코드에 하드코딩이 존재하는가?

**결과 기록 형식:**
```
날짜: YYYY-MM-DD
참여자: [익명 ID]
시나리오: [1/2/3]
완료 시간: N분
발견된 문제: [기술]
개선 제안: [기술]
```

---

### 테스트 라운드 2: 디자이너 검수 (Figma-코드 일치도)

**대상:** 디자이너 2인 이상
**방법:** Figma 원본 디자인과 구현 결과물을 나란히 비교 (Side-by-side Review)

**검수 항목:**

| 검수 항목 | 허용 오차 |
|-----------|-----------|
| 색상 | 오차 없음 (토큰 기준 100% 일치) |
| 간격 (padding/margin) | ±0px (토큰 그리드 기준) |
| 폰트 사이즈 | 오차 없음 |
| 보더 라디우스 | 오차 없음 |
| 반응형 레이아웃 | 모바일/태블릿/데스크탑 3구간 모두 검수 |

**불일치 발견 시 처리:**
1. GitHub Issue 생성 (라벨: `design-mismatch`)
2. 원인 분류: 토큰 누락 / 토큰 오사용 / 토큰 미정의
3. 해당 스프린트 내 수정 완료 필수

---

### 테스트 라운드 3: AI 에이전트 출력 품질 검증

**목적:** AI가 `DRAGME.md` 원칙에 따라 올바른 토큰 기반 코드를 생성하는지 검증

**테스트 케이스:**

```
TC-01: "Primary 색상의 버튼 컴포넌트를 만들어줘"
  기대 출력: tokens.colors.primary 참조 코드
  실패 조건: '#718096' 하드코딩 존재

TC-02: "카드 컴포넌트에 md 간격 padding을 적용해줘"
  기대 출력: tokens.spacing.md 참조 코드
  실패 조건: '16px' 하드코딩 존재

TC-03: "에러 메시지 텍스트 컴포넌트를 만들어줘"
  기대 출력: tokens.colors.error 참조 코드
  실패 조건: '#F56565' 하드코딩 존재
```

**합격 기준:** 10개 테스트 케이스 중 10개 통과 (100%)

## 디자인 검토 프로세스

### 정기 리뷰 사이클

```
매주 금요일 오후 — Design Token Review (30분)
  ├── 이번 주 추가/변경된 토큰 확인
  ├── 하드코딩 감지 자동화 결과 검토
  └── 다음 스프린트 토큰 확장 계획

격주 월요일 오후 — Design-Code Alignment Review (1시간)
  ├── Figma 최신 버전과 코드 일치도 비교
  ├── 불일치 항목 GitHub Issue 생성
  └── 우선순위 결정 및 담당자 지정
```

---

### 코드 리뷰 체크리스트 (PR 머지 전 필수)

모든 PR은 아래 항목을 통과해야 합니다.

**디자인 토큰 준수 여부:**
- [ ] 색상 값이 `tokens.colors.*`로만 참조되는가?
- [ ] 간격 값이 `tokens.spacing.*`로만 참조되는가?
- [ ] 폰트 사이즈가 `tokens.typography.*`로만 참조되는가?
- [ ] 하드코딩된 수치(`px`, hex 코드 등)가 0건인가?

**접근성:**
- [ ] 인터랙티브 요소에 `aria-label` 또는 `aria-describedby`가 있는가?
- [ ] 키보드 포커스 순서가 논리적인가?
- [ ] 색상 대비가 WCAG AA 기준(4.5:1)을 충족하는가?

**컴포넌트 구조:**
- [ ] Atomic 계층 원칙을 준수하는가? (atoms → molecules → organisms)
- [ ] `atoms`가 외부 상태에 의존하지 않는가?

### 디자인 토큰 업데이트 절차

Figma에서 디자인 토큰 값이 변경될 때의 처리 흐름:

```
1. 디자이너: Figma에서 값 변경 → 변경 내역을 Slack/이슈로 공유
       │
       ▼
2. 개발자: tokens.ts에서 해당 토큰 값 업데이트
       │
       ▼
3. 자동: 토큰을 참조하는 모든 컴포넌트에 변경 즉시 반영
       │
       ▼
4. 검수: 디자이너가 구현 결과 확인 (Side-by-side Review)
       │
       ▼
5. 완료: PR 머지 → 스테이징 배포 → 최종 승인
```

**단일 토큰 수정으로 전체 UI 반영이 보장되므로, 코드베이스 전체 탐색 없이 1개 파일만 수정합니다.**

---

### 하드코딩 감지 자동화

CI 파이프라인에서 하드코딩된 색상/간격 값을 자동으로 감지합니다.

```bash
# 하드코딩 색상 감지 (hex 코드 직접 사용 탐지)
grep -rn "#[0-9a-fA-F]\{3,6\}" src/components/ --include="*.tsx" --include="*.ts"

# 하드코딩 px 값 감지 (tokens.spacing 외 직접 px 사용 탐지)
grep -rn "style={{" src/components/ | grep -v "tokens\."
```

**감지 결과가 0건이어야 PR 머지가 허용됩니다.**

### 종합 의견

디자인 시스템 토큰 기반으로 모든 컴포넌트가 일관성 있게 구현되었음. `src/styles/tokens.ts` 단일 소스 원칙이 Button, Input, Badge 전 컴포넌트에 걸쳐 일관되게 적용되었으며, 접근성 속성(ARIA) 또한 모든 인터랙티브 요소에 누락 없이 포함되었음.
